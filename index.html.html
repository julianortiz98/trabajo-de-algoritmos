def taylor_seno(x, n):
    resultado = 0.0
    signo = 1
    for k in range(n):                    # n iteraciones
        exponente = 2*k+1
        potencia  = x**exponente              # ← 1 op
        fact      = factorial(exponente)      # ← (2k+1) ops dentro del factorial
        resultado += signo * potencia / fact  # ← 3 ops
        signo *= -1                           # ← 1 op
CF = Σk=0..n-1 [ 1 + (2k+1) + 1 + 1 ] = Σ [2k+4] = 2·n(n-1)/2 + 4n = n² + 3n ≈ n² + 4n
Conteo verificado con el código real: CF = n²+4n. Si se usa factorial incremental: CF = 4n → O(n)
O(n²) Ω(n²) Θ(n²) con factorial completo · O(n) con factorial incremental

2c — ¿Usan los lenguajes estas series?
Reflexión sobre eficiencia y métodos alternativos

Los lenguajes de programación y calculadoras científicas no usan directamente la Serie de Taylor. Los métodos reales son:

CORDIC
COordinate Rotation DIgital Computer. Solo usa sumas y desplazamientos de bits. Usado en hardware (FPU, calculadoras científicas). Converge en ≈20 iteraciones con precisión de 64 bits.
Polinomios de Chebyshev
Minimizan el error máximo en el intervalo (criterio minimax). Usados en libm de GNU / glibc. Logran el mismo error que Taylor con muchos menos términos (≈ 11-15 coeficientes).
Reducción de argumento
Primero se reduce x al rango [0, π/2] usando simetrías: sin(x+2π) = sin(x), sin(π-x) = sin(x). Luego se aplica un polinomio corto. Esto evita la cancelación numérica de Taylor para x grande.
¿Es eficiente Taylor?
Para |x| pequeño converge rápido. Para |x| grande necesita muchos términos y sufre cancelación numérica (suma de grandes cantidades alternadas). En producción se prefieren métodos O(1) con constante fija.
Código Python — serie de Taylor
def factorial(k):
    resultado = 1
    for i in range(2, k+1): resultado *= i
    return resultado

def taylor_seno(x, n):
    """sen(x) = x - x³/3! + x⁵/5! - ...  (n términos). CF ≈ n²+4n → Θ(n²)"""
    resultado, signo = 0.0, 1
    for k in range(n):
        exp = 2*k+1
        resultado += signo * (x**exp) / factorial(exp)
        signo *= -1
    return resultado

def taylor_coseno(x, n):
    """cos(x) = 1 - x²/2! + x⁴/4! - ...  (n términos). CF ≈ n²+4n → Θ(n²)"""
    resultado, signo = 0.0, 1
    for k in range(n):
        exp = 2*k
        pot  = x**exp if exp > 0 else 1
        fact = factorial(exp) if exp > 0 else 1
        resultado += signo * pot / fact
        signo *= -1
    return resultado

# Porcentaje de error
def pct_error(real, approx):
    return abs(real - approx) / abs(real) * 100 if real != 0 else 0

3c — Porcentaje de error en la evaluación de polinomios
Análisis de error — evaluación exacta vs. punto flotante
Nota sobre el literal c del enunciado: El enunciado hace referencia a la Serie de Taylor en este punto por continuidad con el punto 2. Sin embargo, la evaluación de un polinomio es una operación diferente: no es una aproximación, sino un cálculo exacto.
Con coeficientes enteros → error = 0%
La suma y resta de polinomios con coeficientes enteros (como f(n) = 8n⁴+2n³+5n+9) produce resultados exactos en aritmética de enteros. No existe error de aproximación.

f(2) = 8·16+2·8+5·2+9 = 128+16+10+9 = 163 (exacto)
Con coeficientes flotantes → error IEEE 754
Si los coeficientes son números reales (flotantes), aparece error de redondeo de punto flotante doble precisión:

ε_máq ≈ 2.22 × 10⁻¹⁶

Este error es despreciable para polinomios de grado bajo. A medida que el grado crece, el error de redondeo acumulado puede aumentar, pero nunca tiene la naturaleza de la aproximación de Taylor.
Código Python — arreglo de registros
class Termino:
    """Registro: representa un término coef·xᵉˣᵖ"""
    def __init__(self, coef, exp):
        self.coef = coef
        self.exp  = exp

class Polinomio:
    def __init__(self, grado):
        self.grado    = grado
        self.terminos = [Termino(0, i) for i in range(grado+1)]  # arreglo de registros

    def set_coeficiente(self, exp, coef):
        self.terminos[exp].coef = coef

    def evaluar(self, a):                # CF ≈ n²/2 → Θ(n²)
        resultado = 0.0
        for i in range(self.grado+1):
            potencia = a**i
            resultado += self.terminos[i].coef * potencia
        return resultado

def sumar_polinomios(p, q):           # CF = 3·(max_grado+1) → Θ(n)
    grado_max = max(p.grado, q.grado)
    res = Polinomio(grado_max)
    for i in range(grado_max+1):
        coef_p = p.get_coeficiente(i)
        coef_q = q.get_coeficiente(i)
        res.set_coeficiente(i, coef_p + coef_q)
    return res

4e — ¿Existen infinitos números perfectos?
Análisis matemático
Conclusión: A partir de los resultados computacionales NO se puede concluir que existen infinitos números perfectos. Solo se pueden encontrar los que ya existen en el rango analizado.

Teorema de Euler-Euclides: Todo número perfecto par tiene la forma 2p-1(2p−1) donde 2p−1 es un primo de Mersenne. La infinitud de perfectos pares es equivalente a la existencia de infinitos primos de Mersenne — un problema abierto en matemáticas.

Solo se conocen 51 números perfectos (todos pares, al 2026). Ningún número perfecto impar se ha encontrado ni descartado (se sabe que si existe, es mayor que 101500).

def algoritmo_misterio(n):
    if n % 2 == 0:              # n PAR → un solo bucle
        for i in range(n):
            print(i)              # n operaciones básicas
    else:                        # n IMPAR → bucle doble
        for i in range(n):
            for j in range(n):
                print(i, j)       # n² operaciones básicas
f(n) cuando n es par:
Un bucle de n iteraciones → f(n) = n
Ejemplos: f(2)=2, f(4)=4, f(6)=6, f(10)=10
Θ(n) para entradas pares
f(n) cuando n es impar:
Doble bucle de n² iteraciones → f(n) = n²
Ejemplos: f(1)=1, f(3)=9, f(5)=25, f(11)=121
Θ(n²) para entradas impares
¿Por qué Θ(n) o Θ(n²) serían incorrectos como descripción general?

Para que existiera Θ(g), debería cumplirse c₁·g(n) ≤ f(n) ≤ c₂·g(n) para todo n suficientemente grande con las mismas constantes c₁, c₂.

• Si g(n) = n: para n impar, f(n) = n² >> c₂·n → la cota superior no se cumple.
• Si g(n) = n²: para n par, f(n) = n << c₁·n² para n suficientemente grande → la cota inferior no se cumple.

Conclusión: No existe función g(n) tal que f(n) ∈ Θ(g(n)) globalmente. La complejidad oscila entre O(n) y O(n²) según la paridad.

Código Python — QuickSort (implementación nueva)
def particion(arr, bajo, alto):
    """Esquema de Lomuto. Pivote = último elemento."""
    pivote = arr[alto]
    i = bajo - 1
    for j in range(bajo, alto):          # n-1 comparaciones
        if arr[j] <= pivote:               # ← comparación básica
            i += 1
            arr[i], arr[j] = arr[j], arr[i]
    arr[i+1], arr[alto] = arr[alto], arr[i+1]
    return i + 1

def sort_quick(arr, bajo, alto):
    """QuickSort recursivo. Mejor/prom: Θ(n log n) | Peor: O(n²)"""
    if bajo < alto:
        idx = particion(arr, bajo, alto)
        sort_quick(arr, bajo, idx-1)
        sort_quick(arr, idx+1, alto)
